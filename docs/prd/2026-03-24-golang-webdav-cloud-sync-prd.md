# Golang WebDAV Cloud Sync PRD

## 1. 文档信息

- 文档名称：Golang WebDAV Cloud Sync PRD
- 文档版本：v1.0
- 编写日期：2026-03-24
- 参考资料：[Synology Cloud Sync White Paper - Based on DSM 6.1](https://global.download.synology.com/download/Document/Software/WhitePaper/Package/CloudSync/All/enu/Synology_Cloud_Sync_White_Paper-Based_on_DSM_6.1.pdf)
- 目标形态：单机部署、商用版、仅支持 WebDAV

## 2. 背景与目标

Synology Cloud Sync 白皮书定义了一套成熟的同步产品思路：通过统一同步框架，将本地文件系统与远端存储连接起来，结合本地文件事件监听、远端轮询或事件检测、元数据库、一致性校验和客户端加密，实现稳定、可控的数据同步。其核心价值不在于简单上传下载，而在于持续同步、状态可恢复、异常可修复和运维可观测。

本 PRD 面向一个 Go 实现版本的 Cloud Sync 产品，但范围聚焦于 WebDAV 场景，不做多云厂商统一接入。产品目标是构建一个单机部署的商用同步控制器，在本地文件系统和 WebDAV 远端存储之间提供稳定、可控、可观测的同步能力，支持双向同步、单向同步、路径映射、元数据跟踪、一致性校验、端侧加密、限流调度、审计和基础管理界面。

### 2.1 产品目标

1. 稳定同步：支持本地到 WebDAV、WebDAV 到本地、双向同步三种模式，保证文件新增、修改、删除按规则传播。
2. 准实时与可控平衡：本地通过文件事件尽快上行，远端通过轮询发现变更，并允许配置轮询周期、并发度和带宽限制。
3. 商用运维能力：支持多任务、方向控制、路径映射、失败重试、断点恢复、任务暂停恢复、审计日志和运行监控。
4. 安全与一致性：通过本地元数据库保存文件状态和同步游标，支持一致性检查、冲突处理和客户端加密上传。

### 2.2 非目标

1. 第一阶段不支持 Dropbox、Google Drive、S3 等其他云厂商接入。
2. 不做分布式集群和多节点高可用。
3. 不做在线文档格式转换或协同编辑能力。
4. 不依赖远端必须提供事件推送能力，默认按 WebDAV 轮询模型设计。
5. 不做对象级去重、跨任务内容寻址存储和复杂版本管理。

## 3. 产品定位与范围

本产品定位为“单机部署的 WebDAV 云同步控制器”。它运行在单台主机上，负责在本地文件系统与 WebDAV 远端存储之间建立稳定、可控、可观测的同步链路，提供双向/单向同步、任务管理、元数据跟踪、一致性校验、端侧加密、限流调度和审计监控能力。

架构上保留清晰的连接器边界，但第一阶段只实现 WebDAV Connector，不将多厂商接入作为交付目标。这样可以把产品能力集中在 WebDAV 场景下的目录遍历、远端变更探测、属性映射、覆盖语义、网络异常恢复和性能优化上。

## 4. 用户场景与需求范围

### 4.1 核心用户场景

1. 本地工作目录同步到远端 WebDAV
用户在本地目录持续写入文件，系统通过文件事件监听尽快把新增、修改、删除同步到 WebDAV，适用于文档归档、设计稿备份、业务数据异地留存等场景。

2. 远端 WebDAV 数据回流到本地
远端存储由其他系统或人工维护，系统按轮询周期拉取远端目录变化并同步到本地，适用于把 WebDAV 作为外部共享入口，而本地节点作为统一处理或备份节点。

3. 双向协作同步
本地和远端都可能发生文件变更，系统需要基于元数据和规则判断上传、下载、冲突重命名或保留版本，适用于跨地点团队通过 WebDAV 共享资料。

4. 单向归档与保留
用户希望只把本地新增和修改推送到远端，但不希望远端文件因本地删除而被清理。本质上是“上传单向同步 + 禁止删除传播”。

### 4.2 本阶段必须支持

1. 同步方向：支持双向同步、仅上传、仅下载三种模式。
2. 任务模型：支持创建、启动、暂停、恢复、停止、删除和查看状态。
3. 路径映射：支持本地目录与远端目录非同构映射。
4. 变更探测：本地侧通过文件系统事件监听；远端侧通过 WebDAV 目录轮询和属性比对发现变化。
5. 文件操作范围：支持文件和目录的新增、内容修改、重命名、删除同步；无法稳定识别纯重命名时可退化为删除加新增。
6. 一致性校验：支持基于 type、size、mtime 的基础校验；支持可选的 hash 校验增强模式。
7. 冲突处理：至少支持“保留两份并重命名冲突文件”。
8. 可靠性能力：支持失败重试、任务重启恢复、全量重扫修复。
9. 安全能力：支持 HTTPS 传输、凭据安全存储、可选客户端加密上传。
10. 运维能力：支持基础审计日志、任务运行日志、同步统计、失败告警入口和带宽/并发限制。

### 4.3 本阶段不纳入

1. 多 WebDAV 服务自动编排或跨任务联动。
2. 多租户权限体系。
3. 分布式任务调度。
4. 细粒度版本管理和文件历史回溯。
5. WebDAV 服务端能力探测后的高级协商优化，例如深度利用 LOCK 或服务端复制。

## 5. 产品功能需求

### 5.1 连接与认证管理

1. 支持配置 WebDAV endpoint、用户名、密码、根路径、超时和 TLS 校验策略。
2. 支持连接测试，验证认证是否有效、目标路径是否可访问、基础 WebDAV 方法是否可用。
3. 支持连接配置加密存储，明文密码不得直接落盘。
4. 支持连接编辑、禁用、删除；被任务引用时删除需有保护机制。
5. 支持记录服务端能力探测结果，例如 PROPFIND、ETag、Last-Modified、Content-Length 支持情况。

### 5.2 同步任务管理

1. 支持任务创建向导，至少配置任务名称、本地路径、远端路径、同步方向、轮询周期、冲突策略、删除策略、限速策略、是否开启加密、是否开启增强一致性校验。
2. 支持任务生命周期操作：创建、启动、暂停、恢复、停止、删除。
3. 支持任务状态展示：初始化中、运行中、暂停中、失败、重试中、已停止、已完成。
4. 支持任务级别配置独立生效，不同任务可有不同方向、轮询间隔和限流参数。
5. 支持单机多任务并行运行，并允许配置每个任务的并发上限。

### 5.3 变更探测与同步执行

1. 本地侧通过文件系统事件监听捕获新增、修改、删除、重命名事件。
2. 当本地事件丢失、服务重启或监听溢出时，系统必须自动触发补偿扫描。
3. 远端侧按轮询周期执行目录扫描，获取文件列表和属性，并与本地元数据做差异比较。
4. 对支持 ETag 或稳定 mtime/size 的 WebDAV 服务，优先使用远端属性减少下载校验成本。
5. 同步引擎应将变化转换为统一操作，例如 Upload、Download、DeleteRemote、DeleteLocal、RenameConflictLocal。
6. 支持空目录同步策略配置：保留空目录或忽略空目录。
7. 支持大文件传输重试和中断恢复占位设计；若首版无法完整支持续传，需明确降级行为。

### 5.4 元数据与一致性校验

1. 为每个同步对象记录核心属性：任务 ID、相对路径、文件类型、大小、mtime、本地文件标识、远端 ETag/校验值、最近同步时间、最近同步方向、状态版本号。
2. 支持记录任务游标信息，包括上次扫描时间、上次成功轮询时间、最近全量校验时间。
3. 基础一致性校验默认比较 type、size、mtime。
4. 增强一致性校验在条件允许时增加 hash 比较。
5. 支持定期全量对账，修复由于网络抖动、事件遗漏或异常退出导致的状态偏差。
6. 元数据库损坏或缺失时，系统应支持重建索引并重新建立同步基线。

### 5.5 冲突处理与删除策略

1. 当同一路径在同步窗口内本地和远端均发生修改，默认生成冲突副本，不直接覆盖已有内容。
2. 冲突文件命名需可追踪，至少包含来源端和时间戳。
3. 支持删除传播策略：本地删远端、远端删本地、双向删除传播、禁止删除传播。
4. 在单向上传模式下，支持“仅新增和修改，不传播删除”。
5. 在单向下载模式下，支持“仅拉取和更新，不删除本地已有文件”的保守模式。

### 5.6 安全、限流与运维

1. 支持 HTTPS 传输和证书校验配置。
2. 支持客户端加密上传。首版采用“文件内容加密 + 元数据最小泄露”的方案，密码或密钥材料需安全保存。
3. 支持任务级上传/下载带宽限制。
4. 支持任务级并发限制，避免单机资源被单任务耗尽。
5. 支持结构化日志，区分系统日志、任务日志、审计日志。
6. 支持暴露基础监控指标，例如任务状态、扫描耗时、同步成功数、失败数、重试数、队列长度、传输流量。
7. 支持失败告警集成预留，例如 Webhook 或邮件告警接口。

### 5.7 管理界面与 API

1. 提供 Web 管理界面，用于连接管理、任务管理、日志查看、状态监控和参数配置。
2. 提供后端 API，供前端和未来自动化集成调用。
3. 支持任务运行面板，展示当前扫描状态、最近错误、传输速率和累计同步统计。
4. 支持按任务检索失败记录，并支持手动重试。
5. 支持审计关键操作：创建连接、修改连接、创建任务、暂停/恢复任务、删除任务、变更安全参数。

## 6. 核心流程与系统架构要求

本产品采用“单机部署、模块化分层”的架构。对外是一个服务，对内拆分为连接管理、任务管理、同步引擎、WebDAV 连接器、文件监听、远端扫描、元数据存储、调度限流、日志监控几个核心模块。

### 6.1 核心业务流程

#### 6.1.1 任务创建流程

1. 用户在管理界面创建 WebDAV 连接，填写 endpoint、认证信息、远端根路径和 TLS 参数。
2. 系统执行连接测试，探测远端是否支持必需的 WebDAV 方法及关键属性读取。
3. 用户创建同步任务，配置本地路径、远端路径、同步方向、轮询周期、删除策略、冲突策略、限流参数和加密选项。
4. 系统初始化任务元数据，建立本地索引，生成首轮同步计划。
5. 系统进入首次基线同步，完成后任务进入稳定运行态。

#### 6.1.2 本地变更上行流程

1. 文件监听模块持续监听本地目录事件。
2. 事件进入任务操作队列，经去重和短时聚合后形成待执行动作。
3. 同步引擎结合任务配置和元数据库状态，判断应执行上传、远端删除或冲突处理。
4. WebDAV 连接器将统一动作转换为具体 HTTP/WebDAV 请求，例如目录探测、文件上传、属性读取或删除。
5. 操作成功后更新元数据库与运行指标；失败则按重试策略回退并记录错误。
6. 若监听事件不完整或队列溢出，触发补偿扫描修正状态。

#### 6.1.3 远端变更下行流程

1. 远端扫描模块按轮询周期执行目录扫描。
2. WebDAV 连接器通过 PROPFIND 获取目录树及文件属性。
3. 同步引擎将远端快照与元数据库、本地当前状态做三方比较。
4. 对于远端新增或修改，决定下载或生成冲突副本；对于远端删除，按删除策略决定是否删除本地。
5. 执行完成后刷新扫描游标、同步状态和统计指标。
6. 若轮询失败，进入指数退避重试，并保留上次成功状态供后续恢复。

#### 6.1.4 一致性修复流程

1. 系统定时执行全量或分片一致性检查。
2. 比较本地文件系统、远端目录快照和元数据库记录。
3. 对发现的不一致项生成修复动作，例如重新上传、重新下载、补写状态、标记冲突。
4. 修复完成后输出任务级对账结果和异常摘要。

### 6.2 逻辑架构要求

1. API / Web UI 层：负责连接配置、任务管理、状态展示、日志检索和参数修改，不直接执行同步。
2. 应用服务层：封装连接管理服务、任务服务、调度服务、审计服务，对上提供业务 API，对下协调同步引擎和持久层。
3. 同步引擎：负责把本地事件与远端轮询结果转换成统一同步动作，执行动作排序、冲突决策、重试控制和状态推进。
4. WebDAV 连接器：负责认证、请求封装、目录遍历、属性读取、文件上传下载、删除、错误码转换和能力探测。
5. 本地监听与扫描模块：监听本地文件系统变化，并在异常情况下触发目录补偿扫描。
6. 元数据库：存储连接配置、任务定义、文件索引、同步状态、轮询游标、失败记录和审计信息。
7. 调度与执行队列：负责任务并发控制、队列排队、速率限制和重试调度。
8. 可观测性模块：负责日志、指标、健康检查和告警事件输出。

### 6.3 关键设计要求

1. 同步动作标准化：系统内部统一抽象 UploadFile、DownloadFile、DeleteLocal、DeleteRemote、CreateDirLocal、CreateDirRemote、MarkConflict 等动作，避免策略分散。
2. 元数据驱动：同步状态判断必须结合元数据库中的最近一致状态，不能只依赖当前本地和远端快照。
3. 监听与轮询双机制：本地侧以事件驱动为主，但必须有扫描补偿；远端侧以轮询为主，不假设 WebDAV 提供事件推送能力。
4. 异常恢复优先于绝对实时：服务重启、网络闪断、远端超时、部分文件失败时，系统必须能继续推进任务。
5. 加密与同步解耦：客户端加解密逻辑由独立安全模块处理，不侵入任务调度主流程。

### 6.4 建议部署架构

首版建议部署为单二进制服务，内部包含：

1. HTTP API / Web UI 服务端
2. 后台任务调度器
3. 同步执行 worker
4. 本地数据库
5. 指标与日志输出接口

后续演进点包括：

1. 调度器与执行器可拆分
2. 元数据库可从嵌入式数据库迁移到独立数据库
3. WebDAV 连接器可扩展为更多远端协议连接器

## 7. 非功能需求与验收指标

### 7.1 性能要求

#### 7.1.1 同步时效

1. 本地文件变更在正常负载下，应在 5 秒内进入待执行队列。
2. 本地到远端的非大文件上传，在网络正常时应在 30 秒内开始传输。
3. 远端变更发现时延由轮询周期决定，系统应支持 10 秒到 24 小时的可配置轮询间隔。
4. 首版默认轮询周期建议为 10 秒或 30 秒。

#### 7.1.2 扫描与处理能力

1. 单任务应支持至少 10 万文件规模的目录同步管理。
2. 在 10 万文件 / 100 GB 量级下，系统应能完成首次基线扫描并进入稳定同步状态，不出现内存持续增长或任务失活。
3. 任务执行并发、扫描并发和传输并发必须可配置，避免因远端 WebDAV 性能不稳定导致服务被拖垮。

#### 7.1.3 资源控制

1. 系统必须支持任务级上传/下载限速。
2. 系统必须支持任务级最大并发数限制。
3. 在多任务运行时，应保证单任务异常不会导致整个进程 CPU、内存或 I/O 被无限占用。

### 7.2 可靠性要求

#### 7.2.1 任务连续性

1. 服务异常退出或重启后，任务应能基于元数据库恢复到最近一致状态继续执行。
2. 单个文件同步失败不应导致整个任务终止，失败项应进入重试或失败队列。
3. 系统必须支持任务暂停、恢复和手动重试。

#### 7.2.2 数据一致性

1. 默认模式下，系统应基于 type、size、mtime 进行一致性判断。
2. 增强模式下，支持基于 hash 进行更严格校验。
3. 系统必须支持定期全量对账，发现事件遗漏、扫描偏差或状态漂移后自动修复或标记异常。
4. 在冲突场景下，默认不得静默覆盖用户文件，必须保留至少一份可追溯副本。

#### 7.2.3 失败恢复

1. 网络超时、5xx 错误、连接中断、TLS 握手失败等场景下，系统应按重试策略恢复。
2. 重试应支持退避机制，避免对远端 WebDAV 服务造成持续冲击。
3. 当任务进入持续失败状态时，应提供清晰的错误原因和最近失败记录。

### 7.3 安全要求

#### 7.3.1 传输与凭据安全

1. 默认使用 HTTPS 访问 WebDAV。
2. 用户凭据必须加密存储，不得以明文写入数据库、配置文件或日志。
3. 日志中不得输出密码、完整授权头或可直接复用的敏感令牌。

#### 7.3.2 数据加密

1. 系统应支持可选客户端加密上传。
2. 开启加密后，远端存储看到的应为加密后的文件内容，未授权方不能直接读取原文。
3. 密钥材料必须有明确的存储与恢复方案，PRD 中应说明丢失密钥的后果和恢复边界。

#### 7.3.3 权限边界

1. 本地服务进程应只获得必要目录访问权限。
2. Web 管理端应具备基础登录保护能力，避免未授权操作任务和连接配置。
3. 审计日志应记录关键管理行为。

### 7.4 可运维性要求

1. 系统必须提供结构化日志。
2. 必须区分系统日志、任务日志、审计日志。
3. 必须提供基础指标，包括任务状态、成功数、失败数、重试数、队列长度、扫描耗时、传输速率和流量统计。
4. 每个任务必须能查看最近错误、最后成功同步时间、最近扫描时间和当前阶段。
5. 对失败文件必须支持按任务检索，并展示失败原因。
6. 系统应提供健康检查接口，至少区分“进程存活”和“任务子系统可工作”。
7. 轮询周期、并发度、限速、重试次数、超时等关键参数必须可配置。

### 7.5 可维护性要求

1. 系统内部必须模块化，避免 WebDAV 协议细节散落在任务、UI 和数据库层。
2. 核心同步逻辑、状态机、冲突策略和删除策略应可单元测试。
3. 连接器层必须可被 mock，以支持同步引擎的自动化测试。
4. 数据库 schema 变更必须支持版本迁移。

### 7.6 验收指标

#### 7.6.1 功能验收

1. 能成功创建 WebDAV 连接并通过连通性测试。
2. 能创建并运行双向同步、仅上传、仅下载三类任务。
3. 能正确处理新增、修改、删除、目录创建四类基本同步动作。
4. 能在冲突发生时生成冲突副本，而不是静默覆盖。
5. 能在开启“禁止删除传播”后保留目标端历史文件。
6. 能在任务暂停、恢复、服务重启后继续工作。

#### 7.6.2 稳定性验收

1. 在 10 万文件量级目录下运行 24 小时，任务不崩溃、不死锁、不出现明显内存泄漏。
2. 在模拟网络闪断和远端超时场景下，任务能够自动重试并恢复。
3. 在本地事件监听中断后，系统能够通过补偿扫描恢复一致性。

#### 7.6.3 性能验收

1. 在正常网络下，本地小文件变更能在目标时延内被同步。
2. 在指定硬件条件下，多任务并行运行时，系统资源占用保持在可接受区间。
3. 限速与并发配置生效后，传输行为符合预期，不出现失控抢占。

#### 7.6.4 安全验收

1. 凭据不以明文落盘。
2. 敏感信息不出现在日志中。
3. 开启客户端加密后，远端文件不能直接读取原始内容。

## 8. 版本范围、里程碑与交付计划

### 8.1 版本策略

首版采用分阶段交付策略，优先完成可运行、可恢复、可排障的单机 WebDAV 同步产品，再逐步补齐增强能力。建议划分为三个里程碑版本。

### 8.2 M1：核心可运行版本

目标：打通从任务创建到基础同步执行的主链路，形成可演示、可验证的最小可用版本。

范围：

1. WebDAV 连接管理与连通性测试
2. 单任务同步
3. 本地文件监听
4. 远端目录轮询
5. 基于 type、size、mtime 的基础一致性判断
6. 双向同步、仅上传、仅下载三类模式
7. 基础任务状态管理
8. 基础日志与错误展示

交付标准：

1. 能创建任务并完成首次基线同步
2. 能处理新增、修改、删除、目录创建
3. 能在进程重启后恢复任务并继续执行
4. 能通过基础验收用例

### 8.3 M2：商用增强版本

目标：补齐商用部署所需的可靠性、可控性和运维能力。

范围：

1. 多任务并发运行
2. 任务级限速与并发控制
3. 冲突处理与冲突副本命名
4. 删除传播策略与归档模式
5. 审计日志、任务运行日志、基础指标
6. 手动重试、失败记录检索
7. 补偿扫描与全量一致性修复
8. 管理界面完善

交付标准：

1. 多任务下运行稳定，无明显资源失控
2. 网络抖动和远端超时场景下具备自动恢复能力
3. 运维人员可通过界面或日志快速定位任务失败原因

### 8.4 M3：安全与健壮性增强版本

目标：补齐首版商用上线前的安全和边界稳定性要求。

范围：

1. 客户端加密上传
2. 密钥材料保护与恢复说明
3. 增强一致性校验
4. 大文件失败重试策略增强
5. 健康检查与告警接口
6. 数据库迁移与损坏恢复工具

交付标准：

1. 安全验收项全部满足
2. 长时间运行与异常恢复表现稳定
3. PRD 中定义的非功能关键指标达到上线门槛

### 8.5 推荐排期

在团队资源中等、单机产品优先落地的前提下，建议按以下节奏推进：

1. 第 1 至 3 周：M1 需求落地与核心链路实现
2. 第 4 至 6 周：M2 可靠性、运维能力与管理界面增强
3. 第 7 至 8 周：M3 安全与健壮性补齐、验收与发布准备

实际排期可根据团队人数、UI 范围和测试资源调整。

## 9. 风险与依赖

### 9.1 主要风险

1. 不同 WebDAV 服务端对 PROPFIND、ETag、mtime、编码规则和目录遍历行为支持不一致，可能导致兼容性偏差。
2. 远端缺少稳定变更事件能力，轮询模型在大目录下可能带来较高扫描成本。
3. 本地文件事件监听在高频变更或系统压力下可能丢事件，必须依赖补偿扫描兜底。
4. 客户端加密会影响远端可见元数据和部分一致性判断方式，需要在安全性和可管理性之间平衡。
5. 单机模型下，多任务并行容易受 CPU、磁盘和网络瓶颈影响，限流和调度策略是上线前关键风险点。

### 9.2 外部依赖

1. 稳定的 WebDAV 服务端能力与测试环境
2. 本地文件事件通知能力
3. 嵌入式或独立数据库组件
4. TLS 证书与安全存储机制
5. 前端管理界面资源支持

## 10. 成功标准

当以下条件同时满足时，可认为首版产品达到预期目标：

1. 单机环境下稳定支撑 WebDAV 双向和单向同步。
2. 在目标文件规模和运行时长内表现稳定，可恢复、可排障。
3. 具备上线所需的基本安全能力、运维能力和验收证据。
4. 文档、部署、测试和告警机制齐备，能够支撑试点上线。
